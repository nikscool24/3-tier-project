## Setup & Commands

This section explains how to run the project locally or on a server, the commands used, and how environment variables are applied.

---

## Prerequisites
- **Docker** and **Docker Compose** must be installed manually on your machine/server (one‑time setup).  
- A **GitHub Actions self‑hosted runner** needs to be configured manually for automated deployments on the server.  
- Access to **GitHub Secrets and Variables** for environment configuration.

## Automated Environment File Creation (CD Workflow)
In production deployments, the .env file is not created manually.
It is generated automatically during the Continuous Deployment (CD) workflow using values stored in GitHub Secrets and GitHub Variables.

Secrets & Variables in GitHub
Secrets → sensitive values (database passwords, usernames, host ports).

Variables → non‑sensitive configs (container names, image names, internal ports).

CD Workflow Step
The Write .env file step in cd.yaml dynamically creates a fresh .env file on the server.
It injects values from Secrets and Variables into the file.

Example:
POSTGRES_USER=${{ secrets.POSTGRES_USER }}
POSTGRES_PASSWORD=${{ secrets.POSTGRES_PASSWORD }}
POSTGRES_DB=${{ secrets.POSTGRES_DB }}
BACKEND_IMAGE=${{ secrets.BACKEND_IMAGE }}
FRONTEND_IMAGE=${{ secrets.FRONTEND_IMAGE }}

## Docker Compose Integration
Once the .env file is generated, Docker Compose is run with:
docker compose --env-file .env pull
docker compose --env-file .env up -d --force-recreate --pull always


Compose reads the .env file and injects values into container definitions.

Containers are recreated with the latest images and environment values.

## Deployment (Automated via CI/CD)
On every push to main, GitHub Actions:

CI Workflow → Builds and pushes Docker images for backend, frontend, and reverse proxy.

CD Workflow → Runs on the self‑hosted runner, generates a fresh .env file from GitHub Secrets/Variables, and redeploys containers with Docker Compose.

This ensures deployments are:

Consistent → every run uses the same automated process.

Secure → secrets are never exposed in code or repo.

Automated → no manual .env creation or container management needed.

## Summary
Local setup uses Docker Compose with a manually created .env file.

Production setup uses an automated .env file generated by the CD workflow.

Docker Compose orchestrates all services using injected values.

CI/CD ensures builds and deployments are secure, consistent, and automated.
